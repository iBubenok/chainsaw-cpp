# GOV-0005 — Стандарт ведения PUBLIC-репозитория и протокол публикации

## Статус
- Версия: 3.0
- Статус: Approved
- Дата: 2026-01-08

## Назначение

Документ фиксирует **операционный стандарт** работы с публичным репозиторием GitHub.

PUBLIC-репозиторий — это **полноценное зеркало проектной документации**, содержащее:
- всю архитектурную и проектную документацию;
- код, тесты, инфраструктуру;
- процессные документы (отчёты, протоколы, реестры).

PUBLIC **НЕ содержит** внутренних служебных материалов.

Связанные документы:
- `GOV-0004` — правила состава и аудита
- `DEC-0009` — механизм зеркалирования
- `STD-0001` — стандарт документации

---

## 1) Принципы

### 1.1. Полнота документации

PUBLIC показывает **всё, что нужно для понимания проекта**:
- архитектура и решения;
- процесс разработки (этапы, Gate-отчёты);
- код и тесты;
- реестры и трассируемость.

### 1.2. Единственное ограничение — служебные материалы

PUBLIC не содержит **только** внутренних служебных материалов:
- внутренние инструкции и черновики;
- временные рабочие файлы.

Всё остальное — публикуется.

### 1.3. Процессная терминология разрешена

Термины `Step`, `Gate`, `Phase`, `Iteration`, а также идентификаторы `REQ-*`, `ADR-*`, `TST-*` и т.п. — это **стандартная инженерная практика** и они разрешены в PUBLIC.

---

## 2) Механизм публикации

### 2.1. Структура в PRIVATE

Публичный контент хранится в PRIVATE как зеркало:
- `public/` — готовый к экспорту контент
- `public/README.md` → корень PUBLIC как `README.md`
- `public/docs/` → `docs/` в PUBLIC
- `public/cpp/` → `cpp/` в PUBLIC (когда появится код)

### 2.2. Allowlist-экспорт

Экспорт выполняется инструментами из `tools/publish/`:
- `public_manifest.json` — белый список
- `export_public.py` — экспорт
- `audit_public.py` — аудит

### 2.3. Расширение allowlist

При добавлении новых классов артефактов:
1. Добавить файлы/каталоги в `public/`
2. Обновить `public_manifest.json`
3. Выполнить экспорт и аудит
4. При необходимости — обновить `GOV-0004`

## 3) Модель ветвления

### 3.1. Ветки

- `main` — единственная долгоживущая ветка PUBLIC
  - содержит актуальное опубликованное состояние
  - должна оставаться "зелёной" (сборка и тесты проходят)

- Временные ветки для PR:
  - `docs/<topic>` — документация
  - `infra/<topic>` — CI/инфраструктура
  - `code/<topic>` — код и тесты
  - `release/<version>` — подготовка релиза

После merge PR ветка удаляется.

### 3.2. Merge-стратегия

**Squash-merge по умолчанию.** Один PR = один понятный коммит в истории.

Исключения:
- несколько логически независимых изменений;
- сохранение истории для bisect.

## 4) Версионирование и теги

### 4.1. Схема версий

SemVer: `vMAJOR.MINOR.PATCH`
- до функционального паритета 1:1 — версия `v0.x.y`
- после паритета — `v1.0.0` и далее

### 4.2. Когда повышать версию

- **MINOR** — заметная функциональная прибавка
- **PATCH** — небольшие правки, уточнения

### 4.3. Теги и Releases

- Теги ставятся только на `main`
- GitHub Release для значимых вех
- Текст релиза — на русском, описывает результат

---

## 5) Commit message

### 5.1. Общие правила

- **Язык:** русский
- **Стиль:** естественный инженерный
- **Формат:** `<Категория>: <описание>`

Категории:
- `Документация`, `Код`, `Сборка`, `Тесты`, `CI`, `Инфраструктура`, `Лицензии`

### 5.2. Запреты в commit message

Нельзя включать:
- внутреннюю служебную терминологию
- ссылки на внутренние документы

### 5.3. Примеры

```
Документация: пакет архитектуры для согласования
Документация: добавлены Gate-отчёты и протоколы
Документация: обновлены ADR и требования
Сборка: добавлен базовый CMake-каркас
CI: включена сборка в Windows/Linux/macOS
Код: реализован парсер EVTX
```

## 6) PR процесс

### 6.1. Последовательность

1. Подготовить изменения в PRIVATE (в `public/`)
2. Экспорт:
   ```bash
   python tools/publish/export_public.py --dest <путь> --clean
   ```
3. Аудит:
   ```bash
   python tools/publish/audit_public.py --root <путь>
   ```
4. Ручной просмотр изменённых файлов
5. Создать PR в PUBLIC
6. Squash-merge

### 6.2. Шаблон PR

```markdown
**Что изменилось**
- ...

**Зачем**
- ...

**Как проверить**
- ...

**Риски / ограничения**
- ... (или "Не выявлены")
```

### 6.3. Дополнительные требования

Когда в PUBLIC появится код:
- PR не должен ломать сборку/тесты
- изменения зависимостей документируются
- никаких секретов в репозитории

## 7) Роли

### 7.1. Разработчик

- изменения в PRIVATE и подготовка `public/`
- экспорт и аудит
- создание PR в PUBLIC
- поддержание `main` в рабочем состоянии

### 7.2. Ревьюер

- проверка смысла и полноты
- отсутствие служебных материалов
- проверяемость изменений

---

## 8) Применение стандарта

### 8.1. После каждого этапа

1. Обновить `public/` полными результатами
2. Экспорт в рабочую копию PUBLIC
3. Аудит
4. Подготовить commit message и PR
5. Опубликовать через PR

### 8.2. Что публикуется

- **ВСЯ** документация (включая Gate-отчёты, реестры, протоколы)
- Код и тесты
- CI/CD конфигурации
- Инструменты (кроме `tools/publish/` — остаётся внутренним)

---

## 9) Чек-листы

### 9.1. Перед экспортом

- [ ] Изменения находятся в `public/`
- [ ] Нет служебных материалов
- [ ] Документы соответствуют STD-0001

### 9.2. Перед PR

- [ ] Экспорт выполнен по allowlist
- [ ] `audit_public.py` завершился PASS
- [ ] Commit message корректный
- [ ] Ручной просмотр изменений

### 9.3. После merge

- [ ] PR merged в `main`
- [ ] Ветка удалена
- [ ] Тег поставлен (если релизная веха)
- [ ] Зафиксировано в PRIVATE что опубликовано
