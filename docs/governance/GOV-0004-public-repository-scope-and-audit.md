# GOV-0004 — Публичный репозиторий: состав, очистка и аудит

## Статус
- Версия: 4.0
- Статус: Approved
- Дата: 2026-01-08

## Назначение

Документ фиксирует правила для **публичного репозитория** (GitHub) проекта портирования Chainsaw (Rust) → C++.

Публичный репозиторий — это **полноценное зеркало проектной документации и результатов разработки**:
- содержит всю проектную и архитектурную документацию;
- содержит код, тесты, инфраструктуру;
- **НЕ содержит** внутренних служебных материалов.

Документ описывает:
1. какие классы материалов допускаются в PUBLIC;
2. что категорически исключается (служебные материалы);
3. критерии очистки и соответствие стандартам документации;
4. метод аудита перед публикацией.

---

## 1) Определения

### 1.1. Репозитории

- **PRIVATE** — рабочее пространство проекта: полный набор материалов включая служебные артефакты.
- **PUBLIC** — публичное зеркало результатов, содержащее всю проектную документацию без служебных материалов.

### 1.2. Очистка (sanitization)

**Очистка** — приведение материалов к виду, допустимому для PUBLIC. Все публикуемые материалы должны выглядеть как результат работы **опытного разработчика/архитектора**.

**Принцип очистки:**
1. **Переработка в приоритете:** если документ можно переработать (очистить от служебных признаков) — это ОБЯЗАТЕЛЬНО нужно сделать.
2. **Исключение — крайняя мера:** удаление документа из PUBLIC допускается ТОЛЬКО если переработка невозможна.
3. **Запрет на "удаление вместо очистки":** нельзя просто удалять документы, которые можно очистить.

**Очистка включает:**
- удаление служебных материалов и ссылок на них;
- замену специфичной терминологии на нейтральную инженерную;
- приведение документации к стандарту `STD-0001`.

**Очистка НЕ включает:**
- сокращение документации (полнота важна);
- удаление процессной терминологии (Step, Gate, идентификаторы — разрешены).

### 1.3. Критерий готовности

Документ готов к публикации, если **любой внешний читатель**, изучив его, сделает вывод: "это результат работы опытного инженера".

## 2) Классы артефактов и правила публикации

### 2.1. ПУБЛИКУЕТСЯ (разрешено и рекомендуется)

#### A. Проектная документация

| Категория | Что публикуется |
|-----------|----------------|
| `docs/requirements/` | Постановка задачи, архитектурные требования |
| `docs/architecture/` | To-Be архитектура, технологические матрицы |
| `docs/adr/` | Architecture Decision Records |
| `docs/planning/` | WBS, планы, оценки |
| `docs/guide/` | Руководства разработчика |
| `docs/as_is/` | Анализ As-Is (CLI, фичи, архитектура, данные) |
| `docs/baseline/` | Фиксация baseline upstream |
| `docs/tests/` | Тестовые матрицы |

#### B. Управленческая документация

| Категория | Что публикуется |
|-----------|----------------|
| `docs/governance/` | Стандарты и правила проекта (GOV-*) |
| `docs/decisions/` | Проектные решения (DEC-*) |
| `docs/reports/` | Отчёты, протоколы Gate, верификации |
| `docs/standards/` | Стандарты документации (STD-*) |
| `docs/licensing/` | Лицензии и политики third-party |

#### C. Реестры и трассируемость

| Файл | Что публикуется |
|------|----------------|
| `00_PROJECT_STATE.md` | Текущее состояние проекта |
| `00_ARTIFACT_REGISTRY.md` | Реестр артефактов |
| `00_RISK_REGISTER.md` | Реестр рисков |
| `00_DECISION_LOG.md` | Журнал решений |
| `00_TRACEABILITY.md` | Матрица трассируемости |

#### D. Код и инфраструктура

| Категория | Что публикуется |
|-----------|----------------|
| `cpp/` | Исходный код порта, тесты, конфиги сборки |
| `tools/` | Утилиты для сборки/тестирования/harness |
| `.github/` | CI/CD конфигурации |
| `third_party/` | Вендорированные зависимости и лицензии |

#### E. Тестовые данные

- Минимальные тестовые фикстуры (при наличии лицензионной возможности)
- Инструкции получения данных из публичных источников
- Контрольные суммы для верификации

### 2.2. ИСКЛЮЧАЕТСЯ (категорический запрет)

**Запрещено публиковать внутренние служебные материалы:**

#### A. Служебные файлы

| Категория | Причина исключения |
|-----------|-------------------|
| Внутренние инструкции | Служебные документы |
| Черновые заметки | Рабочие материалы |
| Временные файлы | Не являются результатом |

#### B. Служебные каталоги

| Путь | Причина |
|------|---------|
| `.local/` | Локальные артефакты |
| `build/`, `out/` | Артефакты сборки |
| `inputs/raw/`, `inputs/extracted/`, `inputs/tmp/` | Временные данные |
| `.git/` | Служебные данные git |
| `golden/` | Эталонные прогоны (большие бинарные данные) |

### 2.3. Процессная терминология (РАЗРЕШЕНА)

Следующие термины являются **стандартной частью процесса разработки** и **разрешены** в PUBLIC:

- `Step N` — этапы плана работ
- `Gate X→Y` — контрольные точки
- `Phase`, `Iteration`, `Sprint`, `Milestone`
- Идентификаторы: `REQ-*`, `ADR-*`, `DEC-*`, `MOD-*`, `TST-*`, `ART-*`, `RISK-*`, `GOV-*`, `REP-*`, `RUN-*`

Эти термины отражают стандартную инженерную практику.

## 3) Критерии очистки

### 3.1. Контентные проверки

| Проверка | Критерий |
|----------|----------|
| Служебные файлы | Нет служебных файлов |
| Служебные каталоги | Нет служебных путей |
| Служебные ссылки | Нет ссылок на внутренние материалы |

### 3.2. Соответствие стандартам

Документы должны соответствовать `STD-0001-documentation-standards.md`:
- корректная структура (заголовок, статус, назначение)
- правильное форматирование Markdown
- корректные ссылки на ID
- отсутствие дублирования SoT-информации

### 3.3. Чек-лист перед публикацией

- [ ] Нет служебных файлов
- [ ] Нет служебных каталогов
- [ ] Документы соответствуют STD-0001
- [ ] Все ссылки на ID корректны
- [ ] Лицензии third-party присутствуют

## 4) Метод аудита

### 4.1. Последовательность

1. **Экспорт по allowlist:**
   - применить `tools/publish/export_public.py`
   - экспортировать только разрешённые файлы

2. **Автоматический аудит:**
   - запустить `tools/publish/audit_public.py`
   - проверка служебных файлов и каталогов

3. **Ручной просмотр:**
   - README и ключевые документы
   - список файлов на предмет случайных включений
   - **смысловой анализ** документов на предмет служебных признаков

4. **Проверка стандартов:**
   - соответствие STD-0001
   - корректность ссылок

### 4.2. Результат аудита

- **PASS** — публикация разрешена
- **FAIL** — требуется исправление

Результат фиксируется в PRIVATE (не публикуется).

---

## 5) Эволюция PUBLIC

### 5.1. Принцип пополнения

PUBLIC пополняется **полными результатами** каждого этапа:
- документация добавляется целиком, а не в сокращённом виде
- процессные документы (Gate-отчёты, протоколы) публикуются
- чем выше детализация, тем лучше

### 5.2. Ожидаемое содержание

По мере развития проекта в PUBLIC появляются:
- полный пакет архитектурной документации
- код и тесты порта
- CI/CD инфраструктура
- отчёты верификации и Gate-протоколы
- руководства и инструкции

## 6) Риски и митигация

| Риск | Митигация |
|------|-----------|
| Случайное включение служебных материалов | Автоматический аудит + allowlist экспорт |
| Неполнота документации | Стандарт STD-0001 + ревью |
| Утечка через commit message | Ручная проверка PR |
