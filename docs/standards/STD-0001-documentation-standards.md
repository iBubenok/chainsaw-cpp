# STD-0001 — Стандарт ведения проектной документации

## Статус
- Версия: 1.0
- Статус: Approved
- Дата: 2026-01-08

## Назначение

Документ фиксирует **стандарт ведения проектной и архитектурной документации**, основанный на индустриальных best practices:
- IEEE 1016 (Software Design Description)
- ISO/IEC/IEEE 42010 (Architecture Description)
- arc42 (Architecture Documentation Template)
- ADR (Architecture Decision Records)
- RFC 2119 (Key words for use in RFCs)

Стандарт применяется ко всей документации проекта и обеспечивает:
- полноту и детализацию;
- структурированность и навигируемость;
- трассируемость и проверяемость;
- профессиональный инженерный уровень.

---

## 1) Принципы документирования

### 1.1. Полнота важнее краткости

**Эффективность документации = Полнота + Точность + Структурированность**, а НЕ краткость.

Документация должна содержать достаточно информации для:
- понимания контекста и принятых решений;
- воспроизведения процесса разработки;
- верификации результатов;
- onboarding новых участников.

### 1.2. Документируй решение, а не только результат

Для каждого значимого решения фиксируется:
- **Контекст** — почему возникла необходимость в решении;
- **Варианты** — какие альтернативы рассматривались;
- **Решение** — что выбрано и почему;
- **Последствия** — что это означает для проекта.

### 1.3. Single Source of Truth (SoT)

Каждый класс информации имеет **ровно одно** место, где она формулируется как истина. Остальные документы **только ссылаются** на неё по ID.

### 1.4. Трассируемость

Все сущности проекта связаны через систему идентификаторов:
```
REQ-* → ADR-* → MOD-* → TST-* → REP-*
```

## 3) Структура документа

### 3.1. Обязательные секции

Каждый документ должен содержать:

```markdown
# ID — Название документа

## Статус
- Версия: X.Y
- Статус: Draft / Review / Approved
- Дата: YYYY-MM-DD

## Назначение
[Краткое описание цели документа]

## [Основное содержание]
[Структурированное содержание по теме]

## Связанные документы
[Ссылки на связанные документы по ID]
```

### 3.2. Рекомендуемые секции

В зависимости от типа документа:

**Для архитектурных документов:**
- Контекст и границы
- Компоненты и их ответственность
- Потоки данных
- Решения и обоснования
- Ограничения и риски

**Для решений (ADR/DEC):**
- Контекст
- Варианты (с анализом плюсов/минусов)
- Решение
- Последствия
- Критерии проверки

**Для отчётов:**
- Цель проверки
- Методология
- Результаты
- Выводы и рекомендации

## 4) Форматирование

### 4.1. Markdown

Вся документация ведётся в формате **Markdown** (GitHub Flavored Markdown).

### 4.2. Заголовки

```markdown
# Заголовок документа (H1) — только один
## Основные разделы (H2)
### Подразделы (H3)
#### Детализация (H4)
```

### 4.3. Списки и таблицы

- Использовать нумерованные списки для последовательностей
- Использовать маркированные списки для перечислений
- Использовать таблицы для структурированных данных

### 4.4. Код и команды

```markdown
Inline код: `имя_функции`
Блок кода: тройные обратные кавычки с указанием языка
```

### 4.5. Ссылки на ID

При ссылке на другие сущности использовать формат:
- `REQ-FR-0001` — требование
- `ADR-0001` — архитектурное решение
- `MOD-0001` — модуль
- `TST-0001` — тест
- `см. docs/path/file.md` — ссылка на документ

## 5) Язык и терминология

### 5.1. Язык документации

- **Документы проекта** — на русском языке
- **Технические идентификаторы** — на английском (общепринятый стиль)
- **Код и CLI** — на английском (воспроизведение baseline)

### 5.2. Терминология RFC 2119

Для требований использовать ключевые слова:
- **MUST / ДОЛЖЕН** — обязательное требование
- **SHOULD / СЛЕДУЕТ** — рекомендация
- **MAY / МОЖЕТ** — опциональная возможность

### 5.3. Глоссарий

Специфические термины проекта поясняются при первом использовании или выносятся в глоссарий документа.

## 6) Идентификаторы и нумерация

### 6.1. Формат ID

Базовый формат: `PREFIX-NNNN`
- `PREFIX` — латиница в верхнем регистре
- `NNNN` — 4 цифры с ведущими нулями

### 6.2. Стандартные префиксы

| Префикс | Сущность |
|---------|----------|
| `REQ-*` | Требование |
| `ADR-*` | Architecture Decision Record |
| `DEC-*` | Раннее проектное решение |
| `MOD-*` | Модуль/компонент |
| `TST-*` | Логический тест |
| `ART-*` | Артефакт |
| `RISK-*` | Риск |
| `GOV-*` | Governance документ |
| `STD-*` | Стандарт |
| `REP-*` | Отчёт |

### 6.3. Именование файлов

Формат: `PREFIX-NNNN-краткое-описание.md`

Примеры:
- `ADR-0001-build-system-and-cpp-standard.md`
- `GOV-0001-traceability-and-sot.md`
- `GUIDE-0001-developer-guide.md`

## 7) Версионирование документов

### 7.1. Схема версий

- **Major.Minor** (например, 1.0, 1.1, 2.0)
- Major — существенные изменения структуры/содержания
- Minor — уточнения и дополнения

### 7.2. Статусы документов

| Статус | Описание |
|--------|----------|
| `Draft` | Черновик, в разработке |
| `Review` | На рассмотрении |
| `Approved` | Утверждён |
| `Deprecated` | Устарел, заменён другим документом |

---

## 8) Реестры и трассируемость

### 8.1. Центральные реестры

- — текущее состояние проекта
- — реестр артефактов
- — реестр рисков
- — журнал решений
- — матрица трассируемости

### 8.2. Правила трассируемости

1. Каждое требование (`REQ-*`) должно иметь:
 - связь с архитектурным решением (`ADR-*`) или обоснование
 - связь с тестом (`TST-*`) для верификации

2. Каждый модуль (`MOD-*`) должен иметь:
 - связь с требованиями, которые он реализует
 - связь с тестами, которые его проверяют

3. Каждое решение (`ADR-*/DEC-*`) должно иметь:
 - ссылку на контекст (требования, риски)
 - ссылку на затронутые компоненты

## 9) Качество документации

### 9.1. Критерии качества

| Критерий | Описание |
|----------|----------|
| **Полнота** | Вся необходимая информация присутствует |
| **Точность** | Информация соответствует реальности |
| **Актуальность** | Документ отражает текущее состояние |
| **Структурированность** | Логичная организация материала |
| **Навигируемость** | Легко найти нужную информацию |
| **Трассируемость** | Прослеживаются связи между сущностями |
| **Проверяемость** | Утверждения можно верифицировать |

### 9.2. Чек-лист ревью документа

- Заголовок и статус заполнены корректно
- Назначение документа понятно
- Структура логична и последовательна
- Все термины определены или очевидны
- ID-ссылки корректны и ведут на существующие документы
- Нет противоречий с другими документами
- Нет дублирования информации из SoT-документов

## 10) Процессная терминология

### 10.1. Допустимые термины

Следующие термины являются **стандартной частью процесса разработки** и допускаются в документации:

| Термин | Назначение |
|--------|------------|
| `Step N` | Этап/шаг плана работ |
| `` | Контрольная точка перехода между фазами |
| `Phase` | Фаза проекта |
| `Iteration` | Итерация разработки |
| `Sprint` | Спринт (если применимо) |
| `Milestone` | Веха проекта |
| `Backlog` | Бэклог задач |

### 10.2. Использование в документах

Ссылки на этапы и гейты в документах:
- `` — в заголовке документа для трассируемости
- `см. ` — ссылка на контрольную точку
- `создано на -18` — указание на происхождение
