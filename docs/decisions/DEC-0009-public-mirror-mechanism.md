# DEC-0009 — Механизм PUBLIC/PRIVATE и протокол подготовки публикации

## Статус
- Версия: 3.0
- Статус: Approved
- Дата: 2026-01-08

## Контекст

Нужно подготовить **публичный репозиторий на GitHub** для публикации проектной документации, архитектуры и результатов разработки.

Ключевое ограничение: публичный репозиторий должен содержать **полную проектную документацию**, но **НЕ должен** содержать внутренних служебных материалов.

Рамки состава PUBLIC и критерии очистки определены в:
- `GOV-0004` — состав и аудит
- `GOV-0005` — операционный стандарт
- `STD-0001` — стандарт документации

## Требования к механизму

1. **Полнота публикации:** PUBLIC содержит всю проектную документацию (архитектура, решения, отчёты, реестры, код).

2. **Единственное исключение — служебные материалы:** запрещены только внутренние служебные материалы.

3. **Процессная терминология разрешена:** термины Step, Gate, фазы, идентификаторы REQ-*/ADR-*/TST-* — стандартная инженерная практика.

4. **Минимальный риск утечек служебных материалов:** перенос в PUBLIC только по белому списку путей + автоматический аудит.

5. **Воспроизводимость:** одинаковый результат экспорта при одинаковом состоянии PRIVATE.

6. **Масштабируемость:** поддержка документации, кода, CI, тестов.

## Варианты

### Вариант A — Два независимых репозитория + экспорт по allowlist (выбранный)

**Идея:**
- PRIVATE репозиторий — полное рабочее пространство
- PUBLIC репозиторий — отдельный git-репозиторий на GitHub
- В PRIVATE хранится:
  - публичный контент в `public/`
  - инструменты экспорта в `tools/publish/`
  - внутренние стандарты GOV-0004, GOV-0005

**Плюсы:**
- Низкий риск случайной публикации служебных материалов
- Публичная история полностью отделена от PRIVATE
- Не требуется переписывать историю
- Простота и воспроизводимость

**Минусы:**
- Нужна дисциплина: публичные документы ведутся в `public/`
- Требуется явное расширение allowlist при добавлении новых классов артефактов

### Вариант B — Генерация PUBLIC через фильтрацию git-истории

**Идея:** вести один PRIVATE репозиторий, генерировать PUBLIC через `git filter-repo`.

**Минусы:**
- Высокий риск утечки служебных материалов через историю
- Сложность эксплуатации

### Вариант C — PUBLIC как поддерево внутри PRIVATE

**Идея:** вести `public_repo/` внутри PRIVATE, публиковать через subtree split.

**Минусы:**
- Дублирование
- Риск случайного включения служебных материалов

## Решение

Принять **вариант A**: два независимых репозитория + экспорт по белому списку.

## Реализация в PRIVATE

### Структура

```
PRIVATE/
├── public/           # Публичный контент (зеркало PUBLIC)
│   ├── README.md
│   ├── docs/         # Вся документация
│   ├── cpp/          # Код (когда появится)
│   └── ...
├── tools/publish/    # Инструменты экспорта (не публикуются)
│   ├── public_manifest.json
│   ├── export_public.py
│   └── audit_public.py
├── docs/             # Документация (source)
└── ...
```

### Инструменты

- `tools/publish/public_manifest.json` — белый список экспорта
- `tools/publish/export_public.py` — экспорт в PUBLIC
- `tools/publish/audit_public.py` — аудит на служебные материалы

### Workflow

1. Документация готовится в `docs/` (PRIVATE)
2. Копируется в `public/docs/` (без служебных материалов)
3. Экспорт: `python tools/publish/export_public.py --dest <PUBLIC> --clean`
4. Аудит: `python tools/publish/audit_public.py --root <PUBLIC>`
5. Ручной просмотр
6. Commit/PR в PUBLIC

## Последствия

1. Любой публичный коммит: **экспорт → аудит → просмотр → коммит/PR**.

2. Allowlist — единственный механизм безопасности публикации.

3. При добавлении новых классов артефактов — расширение manifest.

4. Процессные документы (Gate-отчёты, реестры, протоколы) — публикуются.

5. Служебные материалы — никогда не публикуются.

## Критерии проверки

Экспорт создаёт дерево PUBLIC, которое:

**Содержит:**
- Всю документацию (docs/)
- Реестры (00_*.md)
- Gate-отчёты и протоколы
- Код и тесты (когда появятся)
- CI конфигурации (когда появятся)

**НЕ содержит:**
- служебных файлов
- `.local/`, `build/`, `out/`, `inputs/raw|extracted|tmp`
- `golden/` (большие бинарные данные)

Аудит (`tools/publish/audit_public.py`) завершается статусом **PASS**.
