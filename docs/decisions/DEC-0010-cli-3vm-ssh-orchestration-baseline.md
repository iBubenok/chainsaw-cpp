# DEC-0010 — Базовый механизм 3‑VM оркестрации по SSH (sync snapshot + exec)

## Контекст
- Требование отдельной серии работ: режим локальной разработки/проверок должен выполняться на 3 целевых ОС (Windows/Linux/macOS) через выделенные VM по SSH.
- В `docs/cli/CLI-0001-cross-platform-cli-protocol-requirements-and-invariants.md` зафиксированы проверяемые требования и варианты синхронизации состояния репозитория (Option A/B/C).
- В репозитории уже присутствует кроссплатформенный детерминированный упаковщик снапшотов: `tools/snapshot/export_snapshot.py`.

## Решение
1) Принять **Option B (ZIP‑snapshot синхронизация)** как базовый способ доставки одного и того же состояния репозитория на все 3 VM:
   - host формирует один ZIP‑снимок дерева репозитория;
   - этот ZIP копируется на каждую VM и разворачивается в стандартный workspace;
   - далее команды выполняются внутри VM.
2) Принять **system ssh/scp + Python stdlib** как базовую зависимость инструментария:
   - оркестрация через `ssh` (exec) и `scp` (upload/download);
   - любые дополнительные зависимости (rsync/paramiko/expect) допускаются только отдельным решением при наличии доказательной необходимости.
3) Зафиксировать минимальные форматы для доказательности:
   - run spec (JSON) и единый отчёт `run.json` в `.local/vm_runs/<RUN_ID>/` (см. `CLI-0003` и `tools/vm/*`).

## Альтернативы
A) **Option A — git‑синхронизация** на каждой VM.
- Плюсы: инкрементальность, привычный контроль состояния через commit SHA.
- Минусы: требуется git и доступность origin/сетевой политики на каждой VM; сложнее гарантировать чистоту рабочего дерева.

B) **Option C — инкрементальная синхронизация по SSH (rsync/robocopy‑style)**.
- Плюсы: потенциально быстрый цикл.
- Минусы: хуже кроссплатформенность (rsync не стандарт для Windows), больше операционной сложности и исключений.

## Обоснование выбора
- Option B минимизирует требования к VM: в базовом варианте VM не обязана иметь git и доступ к origin.
- Детерминированный ZIP‑снимок упрощает доказательство «одинаковый вход на всех 3 VM».
- Существующий инструмент `tools/snapshot/export_snapshot.py` уже реализует детерминированную упаковку с обязательными исключениями, что снижает стоимость внедрения.

## Последствия
- Производительность синхронизации на больших деревьях может быть хуже, чем у git/rsync (стоимость копирования ZIP на каждую VM).
- Для ускорения в будущем возможно внедрение гибридной схемы (например, git‑синхронизация при доступности и политике), но это требует отдельного решения и критериев.
- В репозиторий не добавляются секреты: ожидается использование SSH‑алиасов и внешнего канала передачи доступов.

## Критерии проверки/пересмотра
- Если в практической эксплуатации окажется, что:
  - VM неизбежно требуют git (например, из‑за политики обновлений и скорости), или
  - стоимость копирования ZIP неприемлема,
  то требуется отдельное решение о смене базового варианта синхронизации с доказательством соблюдения CLI‑INV‑0001/0002 и CLI‑SEC‑0001.
